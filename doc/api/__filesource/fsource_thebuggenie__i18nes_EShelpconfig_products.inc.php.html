  2. <b>Visión general del producto</b><br>
  3. Para entender mejor como está hecho un producto, comencemos con un ejemplo, el cual llamaremos <i>"Mi Sistema Operativo"</i>. Buenísimo, un sistema opeartivo.<br>
  4. <i>"Mi Sistema Operativo"</i> viene en varias ediciones diferentes: <i>"Ultimate edition"</i>, <i>"Home edition"</i> y así sucesivamente. Tomemos esto para el ejemplo.<br>
  5. <br>
  6. <i>"Mi Sistema Operativo"</i> está hecho con varios componentes, tales como el  <i>"Kernel"</i>, <i>"Drivers"</i>, <i>"Aplicaciones"</i> y quizás algunos más. Además, la <i>"Ultimate Edition"</i> también tiene un componente de <i>"Aplicaciones Extra"</i>, las cuales no están disponibles en la <i>"Home Edition"</i>. Esto es mejor para los usuarios de Home. Seguro. De todas maneras, esto es lo que se decidió.<br>
  7. <br>
  8. Todavía con nosotros? <b>Excelente!</b> Sigamos.<br>
  9. <br>
  10. <b>Compilaciones</b><br>
  11. A medida que desarrolle <i>"Mi Sistema Operativo"</i>, necesitará liberar algunas versiones aquí y allá, solo para comprobar que todo funciona. Quizás tenga usuarios impacientes, deseosos de comprobar el progreso realizado desde ayer. Estas "versiones" son llamadas "compilaciones".<br>
  12. <br>
  13. Una compilación tiene un número de versión, que nos dice en que parte del ciclo esa compilación fue liberada.<br>
  14. <b>¿Qué se necesita para un producto?</b><br>
  15. Como se puede ver abajo, BUGS 2 soporta todas estas partes. Esto es lo necesario para informar un problema:
  16. <ul>
  17.     <li>Uno o más productos</li>
  18.     <li>Los productos deben tener al menos una edición</li>
  19.     <li>La edición debe tener al menos un componente</li>
  20.     <li>La edición debe tener al menos una liberación (compilación)</li>
  21. </ul>
