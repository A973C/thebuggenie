  2. <b>Basic overview of a product</b><br>
  3. To better get an understanding of how a product is made up, let's start with an example product - we'll call it <i>"My Operating System"</i>. Cool, an operating system.<br>
  4. <i>"My Operating System"</i> comes in several different editions: <i>"Ultimate edition"</i>, <i>"Home edition"</i> and so on. Let's use these two as examples.<br>
  5. <br>
  6. <i>"My Operating System"</i> is made up of several components, such as <i>"Kernel"</i>, <i>"Drivers"</i>, <i>"Applications"</i> and maybe many more. In addition, the <i>"Ultimate Edition"</i> also has an <i>"Extra applications"</i> component, which is not available in the <i>"Home Edition"</i>. This makes it easier for Home users. Sure. Aaanyway - this is what you've decided.<br>
  7. <br>
  8. Still with us? <b>Great!</b> Let's continue.<br>
  9. <br>
  10. <b>Builds</b><br>
  11. As you develop <i>"My Operating System"</i>, you will need to release some versions here and there, just to check how everything works. Maybe you have impatient customers, eager to check out the progress made since yesterday. These "versions" are called "builds".<br>
  12. <br>
  13. A build has a version number, which tells you where in the cycle that build was released.<br>
  14. <br>
  15. <b>What's needed for a product?</b><br>
  16. As you can see above, BUGS 2 supports all these parts. This is what's needed to report issues:
  17. <ul>
  18.     <li>One or more products</li>
  19.     <li>Products must have at least one edition</li>
  20.     <li>The edition must have at least one component</li>
  21.     <li>The edition must have at least one release (build)</li>
  22. </ul> 
  23. So, that's is all fine and dandy - but how do you set up all this stuff?<br>
